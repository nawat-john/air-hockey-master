# แผนสร้าง Air Hockey Bot ที่แข็งแกร่งที่สุด (10Hz)

> เป้าหมาย: บอทเล่น air hockey แบบจำลอง ที่ "มองเห็น + ตัดสินใจ" ทุก 0.1 วินาที (10Hz) เทียบเท่าความเร็วตอบสนองของมนุษย์ และเอาชนะได้แบบสม่ำเสมอ
> สเปกเครื่อง: RTX 3050 (4–8GB VRAM), RAM 12GB, ทำคนเดียว
> สไตล์: hybrid (physics core + RL) เพราะให้ทั้งความนิ่งและความเก่งเชิงกลยุทธ์

---

## 1. Insight หลัก: 10Hz บังคับให้ "ทำนาย" ไม่ใช่ "ตอบสนอง"

ที่ความเร็วลูกพัค ~5 m/s บนโต๊ะยาว ~2 m พัคเดินทางได้ ~50 cm ใน 0.1 วินาที **ระหว่างที่บอทตัดสินใจหนึ่งครั้ง พัคขยับไปครึ่งโต๊ะแล้ว**

ผลที่ตามมา:
- บอทที่ "ไล่ตามตำแหน่งพัคปัจจุบัน" จะแพ้เสมอ เพราะข้อมูลล้าสมัยทันทีที่ลงมือ
- บอทที่แข็งแกร่งต้อง **ทำนายวิถีพัค (รวมการสะท้อนผนัง) แล้วไปดักรอที่จุดตัด** ล่วงหน้า
- นี่คือเหตุผลที่เราใช้สถาปัตยกรรม hybrid: ให้ "ตัวทำนายวิถี" เป็นทั้ง feature ป้อนเข้า policy และเป็น baseline opponent

นี่คือเส้นแบ่งระหว่างบอทกาก ๆ กับบอทเทพ

## 2. สถาปัตยกรรมรวม (Hybrid)

```
                ┌─────────────────────────────┐
   game state ─▶│  Trajectory Predictor (analytic) │──┐  predicted intercept,
   (10Hz)       │  - ทำนายวิถีพัค + สะท้อนผนัง       │  │  time-to-arrival,
                └─────────────────────────────┘  │  aim target
                                                     ▼
                ┌─────────────────────────────┐
                │  RL Policy (SAC / PPO)            │──▶ mallet target velocity
                │  obs = state + predicted features │    (continuous, 10Hz)
                └─────────────────────────────┘
                                                     │
                ┌─────────────────────────────┐  │
                │  Low-level controller (100Hz+)   │◀─┘  ตามเป้าแบบ smooth
                │  clamp speed/accel ให้สมจริง       │
                └─────────────────────────────┘
```

แนวคิด: **physics จัดการสิ่งที่โมเดลง่าย (วิถีพัคเป็นฟิสิกส์ตรง ๆ), RL จัดการสิ่งที่โมเดลยาก** (เมื่อไหร่ควรบุก/ตั้งรับ, มุมและความแรงในการตี, การหลอกคู่ต่อสู้, การจัดการ latency 10Hz)

## 3. Environment Spec

### 3.1 ฟิสิกส์
- โต๊ะ 2D สี่เหลี่ยม มีประตูสองฝั่ง, ผนังด้านข้างสะท้อนแบบยืดหยุ่น
- พัค: ตำแหน่ง/ความเร็ว, แรงเสียดทานต่ำมาก (air hockey), restitution กับผนัง ~0.9
- ไม้ตี (mallet) สองอัน: จำกัดอยู่ในครึ่งโต๊ะของตัวเอง, มี **max speed + max acceleration** (ห้าม teleport — สำคัญต่อความสมจริงและความยุติธรรม)
- การชนพัค–ไม้ตี: ถ่ายโมเมนตัมตามความเร็วสัมพัทธ์

### 3.2 ความถี่ (สำคัญมาก)
- **Physics step: 200–1000 Hz** (substep ละเอียด) เพื่อกันพัค "ทะลุ" ผนัง/ไม้ตี (ใช้ continuous collision detection หรือ substep เล็ก)
- **Agent decision: 10 Hz** = action repeat ทุก 100ms (บอทเลือก action 1 ครั้ง แล้ว low-level controller รันต่อจนถึงรอบถัดไป)

### 3.3 Observation (ป้อน policy ทุก 10Hz)
หลีกเลี่ยง pixel — ใช้ state vector เร็วและเบากว่ามาก:
- ตำแหน่ง+ความเร็วพัค (4)
- ตำแหน่ง+ความเร็วไม้ตัวเอง (4)
- ตำแหน่ง+ความเร็วไม้คู่ต่อสู้ (4)
- **feature ทำนาย**: จุดที่พัคจะตัดเส้นรับของเรา + เวลาที่เหลือ + จุดตัดหลังสะท้อน 1 ครั้ง (~4–6)
- (ทางเลือก) stack 2–3 เฟรมล่าสุดเผื่อให้ policy เดา dynamics เอง

> ป้อน feature ทำนายเข้าไปคือ "คานงัด" ที่ทำให้เรียนรู้เร็วและแข็งแกร่งขึ้นมาก

### 3.4 Action
- **Continuous**: ความเร็วเป้าหมายของไม้ตี (vx, vy) clamp ตาม max speed → เหมาะกับ SAC/PPO
- ทางเลือก: เป้าหมายเป็น "ตำแหน่ง" แล้วให้ controller วิ่งไปหา (เสถียรกว่าตอนเริ่ม)

### 3.5 Reward (หัวใจของความแข็งแกร่ง — ออกแบบให้ดี)
- ยิงประตูได้ : **+10**
- เสียประตู : **−10**
- own goal : **−15** (กันบอทยิงตัวเอง)
- ตีพัคไปทางประตูคู่ต่อสู้ (สัดส่วนความเร็วทิศ +x) : **+เล็กน้อย shaping**
- ครองพัค/ตีโดน : +0.1
- ป้องกัน: อยู่ระหว่างพัคกับประตูตัวเองตอนพัคเข้าหา : +shaping เล็กน้อย
- ปล่อยพัคนิ่งในแดนตัวเองนาน : −เล็กน้อย (กันเฉื่อย)

> เริ่มจาก reward sparse (+10/−10) แล้วค่อยเติม shaping ทีละตัว ตรวจว่าไม่เกิด reward hacking (เช่นบอทตีพัคชนผนังไปมาเพื่อฟาร์ม shaping)

## 4. ตัวทำนายวิถี (Trajectory Predictor) — สูตรลับ

### 4.1 ทำนายตำแหน่งในอนาคต + สะท้อนผนัง (เทคนิค "คลี่กระจก")
แทนที่จะ simulate ทีละ step ใช้เทคนิค **mirror unfolding**: สะท้อนโต๊ะข้ามผนังด้านข้างซ้ำ ๆ ทำให้วิถีที่เด้งหลายครั้งกลายเป็น "เส้นตรง" ในโลกที่คลี่แล้ว แล้วค่อย fold กลับ

- ให้ความกว้างโต๊ะ `W` ตำแหน่ง y ของพัคหลังเวลา `t`:
  - `y_unfolded = y0 + vy * t`
  - fold ด้วย sawtooth: `y = fold(y_unfolded, W)` (สะท้อนไปมาในช่วง [0, W])
- หาเวลา `t*` ที่พัคไปถึงเส้นรับของเรา `x = x_defense` จาก `x0 + vx*t = x_defense`
- แทน `t*` กลับเพื่อได้จุด y ที่ต้องไปดัก → **นี่คือจุดที่ไม้ตีต้องไปยืนรอ**

### 4.2 เล็งยิงเข้าประตู (รวมการเด้ง 1 ครั้ง)
- ยิงตรง: เล็งจุดบนพัคให้เวกเตอร์ออกหลังชนพุ่งเข้ากลางประตูคู่ต่อสู้
- ยิงเด้ง: ใช้ "ภาพสะท้อนของประตู" ข้ามผนังด้านข้าง แล้วเล็งไปที่ภาพสะท้อนนั้น → ได้ลูกแบงค์ช็อตที่หลอกคู่ต่อสู้

### 4.3 ใช้ทำอะไรบ้าง
1. เป็น **feature** ป้อน RL (ดูข้อ 3.3)
2. เป็น **scripted opponent** ระดับเริ่มต้นให้บอท RL ซ้อมด้วย
3. เป็น **baseline** วัดผล: บอท RL ที่ดีต้องเอาชนะ predictor ล้วน ๆ ได้

## 5. การเทรน RL

### 5.1 อัลกอริทึม
- **SAC** (แนะนำ): sample-efficient สำหรับ continuous control, เรียนเร็วบนเครื่องเล็ก
- หรือ **PPO**: เสถียร, ขนาน env ได้เยอะ, จัดการ self-play ง่าย
- ใช้ Stable-Baselines3 / CleanRL เป็นฐาน

### 5.2 Self-play แบบ League (กุญแจสู่ความแข็งแกร่งจริง)
อย่าให้บอทซ้อมกับตัวมันเอง "เวอร์ชันล่าสุด" อย่างเดียว — จะเกิดวงจรเป่ายิงฉุบ (กลยุทธ์ A ชนะ B, B ชนะ C, C ชนะ A) แล้วลืมวิธีเก่า

ทำ **opponent pool / league** สไตล์ AlphaStar ฉบับย่อ:
1. เริ่มซ้อมกับ scripted predictor (ข้อ 4)
2. เก็บ checkpoint ของตัวเองเป็นระยะเข้า pool
3. ตอนเทรน สุ่มคู่ต่อสู้จาก pool (เน้นตัวที่เรายังเอาชนะยาก — PFSP: prioritized fictitious self-play)
4. วัด Elo ภายใน pool เพื่อดูว่าแข็งขึ้นจริง

### 5.3 Curriculum
1. **Phase A**: ฝึกแค่ "ป้องกันประตู" (พัคยิงเข้าหา, ให้รับให้ได้)
2. **Phase B**: เพิ่ม "ตีกลับเข้าแดนคู่ต่อสู้"
3. **Phase C**: เกมเต็ม vs scripted predictor
4. **Phase D**: self-play league เต็มรูปแบบ

### 5.4 Domain Randomization (เพื่อความทนทาน + เผื่อ sim-to-real)
สุ่มในแต่ละ episode: แรงเสียดทาน, restitution, มวลพัค, max speed ไม้ตี, เพิ่ม noise/latency ใน observation (จำลองเซนเซอร์จริง) → บอทจะไม่ overfit ฟิสิกส์ตัวเดียว

## 6. ความพอดีกับฮาร์ดแวร์ (RTX 3050 / RAM 12GB)

- Observation เป็น state vector ~20 มิติ → network เป็น MLP เล็ก (เช่น 256×256) เทรนบน CPU/GPU เล็กสบาย
- ไม่ใช้ pixel → **RAM ใช้น้อยมาก** SAC replay buffer 1M transitions ของ state vector ~ไม่กี่ร้อย MB
- รัน vectorized envs 8–16 ตัวขนานได้ใน RAM 12GB
- VRAM ใช้น้อย (MLP เล็ก) — 3050 รุ่น 4GB ก็เหลือเฟือ
- ความเร็วเทรน: ฟิสิกส์ 2D เขียนเองด้วย numpy/numba เร็วพอ; ถ้าอยากเร็วระดับล้าน step/นาที พิจารณา Brax/MJX (GPU) แต่เกินจำเป็นสำหรับ 2D

## 7. แผนลงมือ (phased — MVP เก่งใน 2 วัน, แล้วไต่ความแข็งแกร่ง)

> หมายเหตุตามจริง: "แข็งแกร่งที่สุด" เป็นงาน iterative ทำ MVP ที่เก่งระดับเอาชนะคนทั่วไปได้ใน ~2 วัน แล้วเฟสถัด ๆ ไปคือดันเพดาน

**วันที่ 1 — Environment + Predictor + ป้องกัน**
- เช้า: เขียนฟิสิกส์ 2D (พัค, ผนัง, ไม้ตี, ประตู) + render ด้วย pygame, เล่นเองด้วยเมาส์ให้ได้ก่อน
- บ่าย: ห่อเป็น Gymnasium env (decision 10Hz, physics substep), เขียน Trajectory Predictor + ทดสอบความแม่นการทำนายเด้งผนัง
- เย็น: Phase A — เทรน SAC ให้ "ป้องกันประตู" ได้ (ง่ายและให้กำลังใจ)

**วันที่ 2 — เกมเต็ม + ชนะ scripted**
- เช้า: ใส่ reward ตีกลับ + ยิงประตู (Phase B–C), เทรน vs scripted predictor
- บ่าย: ตั้ง self-play พื้นฐาน (vs ตัวเองล่าสุด) + เริ่มเก็บ checkpoint pool
- เย็น: ประเมิน vs predictor และ vs ตัวเก่า, อัด GIF, เขียน README

**เฟสไต่เพดาน (ถ้ามีเวลาต่อ)**
- league + PFSP เต็มรูปแบบ → จัด Elo ภายใน
- domain randomization + observation latency/noise
- เพิ่ม action เป็นความเร็ว (จาก position) เพื่อให้ตีแรง/หลอกได้
- (ทะเยอทะยาน) sim-to-real ลงหุ่นจริงด้วยกล้อง + มอเตอร์ XY

## 8. Hyperparameters เริ่มต้น (SAC)

| พารามิเตอร์ | ค่าเริ่มต้น |
|---|---|
| learning rate | 3e-4 |
| batch size | 256 |
| replay buffer | 1e6 |
| gamma | 0.99 |
| tau (soft update) | 0.005 |
| network | MLP [256, 256] |
| entropy coef | auto |
| train freq | ทุก step / gradient step 1 |
| envs ขนาน | 8 |

ปรับ `gamma` ให้พอเหมาะกับ horizon: ที่ 10Hz, 1 วินาที = 10 step ดังนั้น gamma 0.99 ครอบคลุม ~10 วินาที เพียงพอสำหรับการวางแผนใน rally หนึ่ง ๆ

## 9. การวัดผล (รู้ได้ยังไงว่าเก่งขึ้น)

- **Win rate vs scripted predictor** (baseline แรกที่ต้องผ่าน)
- **Elo ภายใน league** (วัดความก้าวหน้าระหว่างเวอร์ชัน)
- **เมตริกย่อย**: % การรับประตูสำเร็จ, ความเร็วเฉลี่ยของลูกที่ยิง, เวลาครองพัค, จำนวน own goal (ต้องเข้าใกล้ 0)
- **vs มนุษย์**: ให้คนเล่นจริงผ่าน pygame แล้วนับ win rate — บททดสอบสุดท้าย

## 10. กับดักที่ต้องระวัง

- **พัคทะลุผนัง/ไม้ตี**: physics step หยาบไป → ใช้ CCD หรือ substep เล็ก ตรวจด้วย test การชนตรง ๆ ก่อนเทรน
- **Reward hacking**: บอทฟาร์ม shaping (เช่นตีพัคชนผนังรัว ๆ) → เริ่ม sparse ก่อน เติม shaping ทีละตัว
- **Self-play collapse**: ซ้อมแค่ตัวล่าสุด → ใช้ league/PFSP
- **เก่งเกินจริงเพราะรู้อนาคต**: ระวังอย่าให้ obs มี info ที่มนุษย์ไม่มี (เช่นความเร็วคู่ต่อสู้แบบ perfect) ถ้าจะ "ยุติธรรม" ให้ใส่ noise/latency
- **10Hz แต่ controller โง่**: low-level controller ต้องตามเป้าได้ smooth ไม่งั้น action ดีแค่ไหนก็เสียเปล่า

## สรุปเส้นทางสู่ "แข็งแกร่งที่สุด"

1. ฟิสิกส์แม่น + ตัวทำนายวิถีที่เด้งผนังได้ = รากฐาน
2. ป้อน feature ทำนายเข้า RL = เรียนเร็วและฉลาด
3. self-play league + PFSP = ไม่ลืมกลยุทธ์ ไต่เพดานได้จริง
4. domain randomization = ทนทาน ไม่เปราะ
5. วัดด้วย Elo + win rate vs มนุษย์ = รู้ว่าเก่งขึ้นจริง

ความเก่งสูงสุดมาจากการ iterate — แต่แค่ถึง "ข้อ 2–3" บอทก็เอาชนะคนเล่นทั่วไปได้สบายแล้ว
